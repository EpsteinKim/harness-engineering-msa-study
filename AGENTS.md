# harness-back Agents

> 이 문서는 harness-back 프로젝트의 에이전트(역할) 정의, 책임 범위, CI/CD 파이프라인에서의 권한을 정의합니다.
> 모든 행동과 결정은 [CONSTITUTION.md](./CONSTITUTION.md)의 원칙을 따릅니다.

---

## 운영 모드

> 이 프로젝트는 1인 실습 프로젝트입니다.
> 모든 에이전트 역할은 운영자 본인이 겸임하며, 역할 분리는 **프로세스 학습 목적**입니다.
> AI 어시스턴트(Claude)는 Developer 역할을 보조하되, 최종 결정은 운영자가 내립니다.

---

## 에이전트 구성

| 에이전트 | 식별자 | 역할 한줄 요약 |
|----------|--------|----------------|
| 👑 CEO | operator | 최종 의사결정 및 프로덕션 배포 승인 |
| 🛠 Developer | operator + claude | 서비스 설계, 구현, 기술 검토 |
| 📋 Planner | operator | 요구사항 정의, 우선순위 결정, 일정 관리 |
| 🔍 QA | operator | 품질 검증, 테스트 설계, 배포 전 승인 |

---

## 에이전트 상세 정의

---

### 👑 CEO

> 프로젝트의 최종 책임자. 방향성과 프로덕션 배포에 대한 최종 권한을 가진다.

**책임 범위**

- 프로젝트 전체 방향성 및 우선순위 최종 결정
- 프로덕션(`prod`) 배포 최종 승인
- CONSTITUTION.md 변경 승인
- 에이전트 역할 변경 승인
- 장애 발생 시 비즈니스 영향도 판단 및 대응 방향 결정

**권한**

| 환경 | 배포 트리거 | 승인 |
|------|-------------|------|
| dev | ❌ | ❌ |
| staging | ❌ | ❌ |
| prod | ✅ 수동 트리거 | ✅ 최종 승인 |

**에스컬레이션 조건**

- prod 장애 발생 시
- 서비스 SLA 위반 시
- 아키텍처 변경이 필요한 의사결정 시

---

### 🛠 Developer

> 서비스의 설계와 구현을 담당한다. 기술적 품질과 일관성에 책임을 진다.
> AI 어시스턴트(Claude)는 이 역할의 보조로 동작하며, DB 스키마 변경 시 반드시 운영자 승인을 받는다.

**책임 범위**

- 서비스 설계 및 구현
- 코드 리뷰 (PR 승인)
- 기술 부채 관리 및 리팩토링
- 인프라/배포 스크립트 관리 (Dockerfile, docker-compose.yml)
- dev 환경 배포 및 운영
- 장애 발생 시 원인 분석 및 핫픽스 작성

**AI 어시스턴트 제약**

| 행위 | 자동 수행 | 승인 필요 |
|------|-----------|-----------|
| 코드 작성/수정 | ✅ | - |
| 테스트 코드 작성 | ✅ | - |
| API 스펙 자동 생성 | ✅ | - |
| DB 스키마 변경 | ❌ | ✅ 운영자 승인 필수 |
| 의존성 추가/변경 | ❌ | ✅ 운영자 승인 필요 |

**권한**

| 환경 | 배포 트리거 | 승인 |
|------|-------------|------|
| dev | ✅ 자동 (PR 머지 시) | ✅ |
| staging | ✅ 수동 트리거 | ❌ (QA 승인 필요) |
| prod | ❌ | ❌ |

**CI/CD 파이프라인 담당 단계**

```
[Build] → [Unit Test] → [Code Quality Check] → [Docker Build] → [Deploy to dev]
   ↑ Developer 담당 구간
```

**에스컬레이션 조건**

- 기술적으로 요구사항 구현이 불가능한 경우 → Planner에게 에스컬레이션
- 품질 기준 미달로 배포 불가 판단 시 → QA와 협의

---

### 📋 Planner

> 요구사항을 정의하고 우선순위를 결정한다. 개발과 비즈니스 사이의 가교 역할을 한다.

**책임 범위**

- 기능 요구사항 문서화 및 관리
- 스프린트/이터레이션 계획 수립
- 개발 우선순위 결정
- 릴리즈 노트 작성
- 배포 일정 관리 및 블랙아웃 타임 조율
- Phase 전환 시점 판단 (Phase 1 → 2 → 3)

**권한**

| 환경 | 배포 트리거 | 승인 |
|------|-------------|------|
| dev | ❌ | ❌ |
| staging | ❌ | ❌ |
| prod | ❌ | ❌ |

> Planner는 배포 권한을 갖지 않는다. 단, 배포 일정과 블랙아웃 타임에 대한 **거부권**을 가진다.

**CI/CD 파이프라인 담당 단계**

```
[Release Note 작성] → [배포 일정 확인] → [블랙아웃 타임 검토]
   ↑ Planner 담당 구간 (prod 배포 전)
```

**에스컬레이션 조건**

- 일정 내 구현 불가 판단 시 → CEO에게 에스컬레이션
- 요구사항 충돌 발생 시 → CEO 중재 요청

---

### 🔍 QA

> 품질 검증의 최종 책임자. staging 환경에서의 배포 승인 권한을 가진다.

**책임 범위**

- 테스트 케이스 설계 및 관리
- staging 환경 검증 및 배포 승인
- 버그 리포트 작성 및 추적
- 회귀 테스트 수행
- 성능/부하 테스트 기준 정의
- 배포 후 smoke test 수행

**권한**

| 환경 | 배포 트리거 | 승인 |
|------|-------------|------|
| dev | ❌ | ❌ |
| staging | ❌ | ✅ 검증 후 승인 |
| prod | ❌ | ✅ 품질 승인 (CEO 최종 승인 전) |

**CI/CD 파이프라인 담당 단계**

```
[Integration Test] → [E2E Test] → [Staging 검증] → [QA 승인] → [Smoke Test (prod 후)]
                                        ↑ QA 담당 구간
```

**에스컬레이션 조건**

- 품질 기준 미달로 staging 승인 거부 시 → Developer에게 재작업 요청
- 반복적인 품질 이슈 발생 시 → CEO에게 보고

---

## CI/CD 전체 흐름 요약

```
 코드 작성 (Developer + Claude)
      ↓
 PR 생성 → 코드 리뷰 (Developer)
      ↓
 CI 파이프라인 자동 실행
   [Build → Unit Test → Code Quality → Docker Build]
      ↓
 dev 자동 배포 (Developer)
      ↓
 staging 수동 트리거 (Developer)
      ↓
 staging 검증 → QA 승인 (QA)
      ↓
 배포 일정 확인 (Planner)
      ↓
 prod 배포 승인 → prod 배포 (CEO)
      ↓
 Smoke Test (QA)
```
